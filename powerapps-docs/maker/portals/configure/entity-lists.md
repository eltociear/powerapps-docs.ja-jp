# <a name="about-entity-lists"></a>エンティティリストについて

エンティティリストは、レコードの一覧を表示する web ページを追加するために使用するデータドリブン構成で、開発者がポータルにグリッドを表示する必要がありません。 エンティティリストを使用すると、ポータルに表示するレコードを公開できます。

グリッドは並べ替えをサポートしており、レコードの数が指定されたページサイズよりも大きい場合は、改ページ調整されます。 **詳細ビューの Web ページ**が指定されている場合、各レコードにはページへのリンクが含まれ、Id クエリ文字列パラメーター名と共に、レコードの id がクエリ文字列に追加されます。 エンティティリストでは、複数のビューもサポートされています。 複数のビューが指定されている場合は、ユーザーがさまざまなビューを切り替えることができるように、ドロップダウンリストが表示されます。

データは、現在のポータルユーザー、現在のポータルユーザーの親カスタマーアカウント、および現在のポータル web サイトでもフィルター処理できます。 フィルター条件**ポータルのユーザー属性**と**アカウント属性**の両方に値が存在する場合、ポータルではドロップダウンリストが表示され、ユーザーは自分の (My) データまたは親顧客アカウントのデータを表示できます。

## <a name="add-an-entity-list-to-your-portal"></a>ポータルにエンティティリストを追加する

エンティティリストには、web ページへのリレーションシップと、ポータル内のレコードの一覧の初期化を制御するさまざまなプロパティが含まれています。 Web ページへのリレーションシップを使用すると、web サイト内の特定のページノードのリスト定義を動的に取得できます。 既存のエンティティビューを表示したり、新しいエンティティビューを作成したりするには、**ポータル** > **エンティティリスト**にアクセスします。

> [!Note]
> - リストをサイト内で表示できるようにするには、エンティティリストが特定の web サイトの web ページに関連付けられている必要があります。
> - 複数選択オプションセットは、エンティティリストではサポートされていません。

エンティティリストに関連付けられている Web ページを表示するには、左端のメニューの**関連する**ナビゲーションリンクに一覧表示されている **[Web ページ]** リンクを選択します。 エンティティの一覧を作成する場合、最初の手順は、ポータルでリストを表示するエンティティを選択することです。 次に、表示するモデル駆動型アプリのビューを1つ以上選択します。

Web ページを作成または編集するときに、Web ページフォームに表示される参照フィールドにエンティティリストを指定できます。 通常、ページテンプレートはページテンプレートになりますが、マスターテンプレートにはエンティティリストをレンダリングする必要があるかどうかを判断するために必要なロジックが含まれているため、コンテンツ用に設計された他のテンプレートのいずれかを使用できます。

## <a name="add-custom-javascript"></a>カスタム Javascript の追加

フォームの [オプション] タブには、カスタム [!INCLUDE[pn-javascript](../../../includes/pn-javascript.md)]を入力できるテキスト領域が含まれています。ページに jQuery library が含まれている場合は、ここでも使用できます。 スクリプトブロックは、ページの終了フォームタグの直前に web ページの下部に追加されます。

![カスタム JavaScript の例](../media/custom-javascript-example.png "カスタム JavaScript の例")  

リストはデータを非同期に取得し、完了すると、カスタム [!INCLUDE[pn-javascript](../../../includes/pn-javascript.md)] がグリッド内の項目をリッスンして処理できるイベント `loaded` をトリガーします。 次のコードは、簡単な例です。
```
$(document).ready(function (){
$(".entitylist.entity-grid").on("loaded", function () {
$(this).children(".view-grid").find("tr").each(function (){
// do something with each row
$(this).css("background-color", "yellow");
});
});
}); 
```

特定の属性フィールドを検索し、その値を取得して、値の表示を変更する可能性があります。 次のコードは、`accountnumber` 属性の値を含む各テーブルセルを取得します。 `accountnumber` を、ご使用のエンティティおよびビューに適した属性に置き換えます。
```
$(document).ready(function (){
   $(".entitylist.entity-grid").on("loaded", function () {
      $(this).children(".view-grid").find("td[data-attribute='accountnumber']").each(function (i, e){
         var value = $(this).data(value);
         // now that you have the value you can do something to the value
      });
   });
});
```

## <a name="securing-entity-lists"></a>セキュリティで保護されたエンティティリスト

エンティティリストをセキュリティで保護するには、レコードが表示されているエンティティのエンティティアクセス許可を構成し、エンティティリストレコードのエンティティの**アクセス許可を有効**にするブール値を true に設定する必要があります。

エンティティリストをセキュリティで保護すると、ページにアクセスするユーザーに対して、アクセス許可が付与されているレコードのみが表示されるようになります。 これは、リストを介して表示されるモデル駆動型アプリビューに追加されるフィルターによって実現されます。 このフィルターは、**読み取り**アクセス許可を使用して、ユーザーがアクセスできるレコードのみをフィルター処理します。

また、リストに対して定義されているアクションは、レコードごとに、そのアクションに対応するアクセス許可を尊重します。 つまり、レコードに対する編集アクセス許可がある場合、そのレコードに対して Edit 操作が有効になります。 Delete、Create などにも同じことが当てはまります。 使用できるレコードがない場合は、一覧が読み込まれたときに、そのことを示すメッセージが表示されることに注意してください。

ただし、優れた web サイトの設計では、エンティティに対するアクセス許可を持つユーザーがロールに含まれていない (つまり、レコードを表示する必要がない) 場合は、ページにアクセスできないようにする必要があります。 Web ページのアクセス許可を使用してページを保護するのが理想的です。

## <a name="adding-a-view-details-page"></a>詳細表示ページの追加

[詳細ビュー] の Web ページを Web ページに設定すると、関連付けられているフォームまたはページの構成に応じて、グリッドに表示されているレコードの詳細を読み取り専用または編集済みとして表示できます。

このページは、通常は液体を使用して作成された、完全にカスタマイズされたページテンプレートにすることができます。 最も一般的なシナリオとして、[詳細] ページが、エンティティフォームまたは Web フォームを含む web ページであることが考えられます。

注意すべき重要な点は、グリッドに表示される各レコードには詳細ページへのハイパーリンクがあり、リンクにはレコードの ID を持つ名前付きクエリ文字列パラメーターが含まれていることです。 クエリ文字列パラメーターの名前は、エンティティリストで指定された ID クエリ文字列パラメーター名によって異なります。 最後に注意すべき点は、対象となる詳細 web ページでも、このクエリ文字列パラメーターの名前を認識して、クエリを実行してデータを読み込む必要があるレコードの ID を取得する必要があることです。

![ビューの詳細ページの追加](../media/add-view-details-page.png "ビューの詳細ページの追加")  

**エンティティフォームを使用して詳細を表示する**

エンティティフォームを作成するには、[エンティティフォーム](entity-forms.md)ページで説明されている手順を参照してください。

エンティティリストのレコードがエンティティフォームに読み込まれるようにするために注意する必要がある重要な設定を次に示します。

エンティティ形式のレコード ID クエリ文字列パラメーター名は、エンティティリストの ID クエリ文字列パラメーター名と一致している必要があります。

モードは、必要に応じて、編集または読み取り専用にすることができます。

**Web フォームを使用して詳細を表示する**

エンティティリストのレコードが Web フォームに読み込まれるようにするために注意する必要がある重要な設定を次に示します。

Web フォームのステップの主キークエリ文字列パラメーター名は、エンティティリストの ID クエリ文字列パラメーター名と一致している必要があります。

モードは、必要に応じて、編集または読み取り専用にすることができます。

**Create 関数の詳細ページの使用**

Create 関数では、カスタムページ、エンティティフォーム、または Web フォームを同じ方法で使用できます。 これは、フォームに対して Create アクションを定義する方法に代わるものです。 Create アクション*と*カスタムページの両方を定義して作成することはできません。カスタムアクションの定義が優先されます。

エンティティの一覧に web ページを割り当て、構成を使用して作成アクションを指定しなかった場合は、[作成] ボタンが一覧に表示されます。このボタンをクリックすると、作成用に指定したカスタムページにユーザーがリンクされます。

## <a name="entity-list-filter-configuration"></a>エンティティリストフィルターの構成

エンティティリストのレコードをフィルター処理する機能を追加するのは簡単です。フィルター処理オプションを有効にし、ユーザーに表示する1つ以上のフィルターの種類を選択するだけです。 ユーザーが指定したテキストに一致する属性を使用してフィルター処理することも、一連のオプションから選択することもできます。 高度な検索を使用して想像できるあらゆる種類のフィルターを設計することもできます。

**エンティティリストフィルターを有効にする**

[メタデータフィルター] セクションで、[有効] チェックボックスをオンにします。 これにより、表示されたときに、フィルター領域がエンティティリストに追加されます。 少なくとも1つのフィルターの種類を定義するまで、ボックスは空の形式で表示されます。

[方向] 設定を使用して、エンティティリストのフィルター領域をどのように表示するかを定義できます。 既定では、[横] を選択すると、フィルター領域がエンティティリストの上に表示されます。 [垂直方向] は、エンティティリストの左側のボックスとしてフィルター領域を表示します。

![メタデータフィルターの設定](../media/metadata-filter-settings.png "メタデータフィルターの設定")  

**フィルターの種類**

|**フィルターの種類**      |**Description**                                                                                                                                                                                                                               |
|----------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| テキストフィルター          | テキストボックスを使用してエンティティリストをフィルター処理し、指定されたエンティティの選択した属性内の一致するテキストを検索します。                                                                                                                               |
| 属性フィルターセット | 一連のチェックボックスを使用してエンティティリストをフィルター処理します。各チェックボックスでは、特定のエンティティの特定の属性に対して条件を一致させようとします。                                                                                           |
| 参照セット           | 一連のチェックボックスを使用してエンティティリストをフィルター処理します。各チェックボックスは、特定のエンティティのレコードと、関連エンティティのレコードとの間のリレーションシップを表します。                                                                         |
| 範囲フィルターセット     | 属性フィルターセットに似ていますが、各チェックボックスは1つではなく2つの条件 (たとえば、0以上で100未満) を表すことができます。                                                                    |
| 動的候補リストセット | 属性フィルターセットの候補リストの値を選択するのと同様です。 動的候補リストセットでは、フィルター処理に使用する候補リストオプションを指定する必要はありません。代わりに、エンティティリストが読み込まれるときにオプションの完全な一覧が生成されます。 |
| 動的参照セット   | 参照セットに似ています。 動的参照セットでは、フィルター処理に使用する参照オプションを指定する必要はありません。代わりに、エンティティリストが読み込まれるときにオプションの完全な一覧が生成されます。                                           |
| FetchXML フィルター      | FetchXML フィルター条件を使用して、エンティティの一覧をフィルター処理します。                                                                                                                                                                                     |

**テキストフィルター**

テキストフィルターは、エンティティリストのエンティティ型の属性に関連付けられているテキストボックスを [エンティティリスト] フィルター領域に追加します。 ユーザーがフィルターを適用すると、エンティティの一覧には、選択した属性に値が含まれているレコードのみが表示されます。

テキストフィルターを追加するには、 **[+ テキストフィルター]** を選択します。

![テキストフィルターを追加する](../media/add-text-filter.png "テキストフィルターを追加する")  

テキストフィルターでは、次の属性を使用します。

|**Name**     |**Description**                                                                                                                                        |

| 属性    | フィルター処理の対象として選択されたエンティティ型の属性の名前。 *Text フィルターでは、文字列型の属性のみが有効です。*                                                                                   |
| 表示名 | エンティティの一覧を表示するときに、フィルターのラベルを上書きします。 既定では、これは選択した属性の名前に自動的に設定されます。 |

**属性フィルターセット**

**属性フィルターセットのオプション**

属性フィルターセットには、通常、任意の数のオプションを指定できます。ただし、候補リストとブール属性は例外です。 ブール属性フィルターセットには、1つまたは2つのオプション&mdash;1 つの true オプションと1つの false オプションを指定できます。 候補リスト属性フィルターセットには、候補リストの各値に対して最大1つのオプションを設定できます。

オプションには次の属性があります。

|**Name**     |**Description**                                                                                                                                                                                  |
|--------------|--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Operator     | 結果をフィルター処理するために使用される比較演算子 (たとえば、Equals、未満など)。 オプションの演算子の一覧は、フィルターに対して選択した属性の種類によって異なります。 たとえば、numeric (Decimal) 型には、より小さい、より大きいなどの演算子があります。一方、文字列属性では、で始まるや Contains などの演算子が使用されます。 候補リストとブール演算子は常にと等しくなります。                                                                                                                                               |
| Value        | このフィルター条件に使用される実際の値。                                                                                                                                                 |
| 表示名 | フィルターボックスで、このオプションの表示名を上書きします。 既定では、この値は Value 属性と同じ値に設定されます。                                                             |

範囲フィルターセットは、フィルター領域に1つまたは2つの条件を持つ一連のオプションを追加します。 ユーザーがフィルターを適用すると、エンティティの一覧には、選択したオプションの少なくとも1つの条件に一致するレコードのみが表示されます。

| オプション      | フィルター処理に使用できる値のコレクション。 詳細については、以下を参照してください。                                                                              |

**動的候補リストセット**

動的候補リストセットでは、指定された候補リストフィールドのすべての値を表す、によってフィルター処理される一連のオプションが追加されます。 これは、属性フィルターセットで候補リストを選択することとは異なります。 属性フィルターセットでは、ユーザーがフィルター処理に使用できるようにする一連のオプションを指定する必要があります。動的候補リストセットでは、候補リストフィールドのみを指定する必要があり、オプションのセット全体が自動的に提供されます。 より高度な制御が必要な場合は、属性フィルターセットを使用することをお勧めします。

## <a name="entity-list-map-view"></a>エンティティリストマップビュー

エンティティリストを使用すると、データのマップビューを有効にして構成することができます。このビューでは、検索機能を使用して、住所の近くの場所を検索することによって [!INCLUDE[pn-bing](../../../includes/pn-bing.md)] します。 緯度と経度の座標値を使用してレコードを設定し、このセクションに記載されている必要な構成オプションを指定することにより、レコードをマップ上でプッシュピンとしてレンダリングできます。 緯度または経度の値を持たないレコードは、検索から除外されます。 ページの初期読み込みでは、既定の中央の緯度座標と既定の中心経度座標から、Distance 値フィールドの初期値 (指定された距離単位に応じてマイルまたは km) 内のすべてのレコードが表示されます。 マップビューが使用されている場合は、指定されたビューが無視されます。また、データセットに距離クエリが適用され、マッピング可能な結果が返されます。
>[!Note] 
>このオプションは、ドイツ語のソブリンクラウド環境ではサポートされていません。 この環境では、[マップビュー] セクションは表示されません。


## <a name="entity-list-calendar-view"></a>エンティティリストカレンダービュー

エンティティリストカレンダービューを使用すると、エンティティリストをカレンダーとして表示できます。個々のレコードは、1つのイベントとして機能するように構成されます。

カレンダーを使用してレコードを表示するには、そのレコードに少なくとも日付フィールドを含める必要があります。 イベントの開始時刻と終了時刻が正確になるようにするには、適切なフィールドを配置する必要があります。 これらのフィールドが構成されていると仮定すると、エンティティリストカレンダービューがポータルに表示されます。

## <a name="enhanced-view-filter-for-entity-lists"></a>エンティティリストの拡張ビューフィルター

有効にすると、エンティティを OData フィードに発行できるようになります。 OData プロトコルは、RESTful web サービスを介してデータと対話するためのアプリケーションレベルのプロトコルです。 このフィードのデータは、web ブラウザーで表示することも、クライアント側の web アプリケーションで使用することも、[!INCLUDE[pn-excel-short](../../../includes/pn-excel-short.md)]にインポートすることもできます。

## <a name="entity-list-odata-feeds"></a>エンティティリスト OData フィード

エンティティのアクセス許可は、レコードをセキュリティで保護する場合に使用できます。ただし、現在のポータルユーザーに関連するフィルターオプションのセットの一部としてフィルターを提供するだけの場合は、エンティティリスト機能を使用できます。 この機能は、現在のユーザー、ユーザーの親アカウント、または web サイトの詳細なフィルター処理をサポートしています。 単に1つの連絡先レコードと一致するようにビューフィルターを作成するだけで、コードはその値を実行時に実際の値に置き換え&mdash;、[フィルター条件] セクションのフィールドに値を割り当てる必要はありません。

> [!Note]
> 公開されている OData フィードは匿名であり、承認チェックが行われていません。そのため、匿名ポータルへのアクセスに適さないデータに対して oData フィードを有効にしないことが重要です。
